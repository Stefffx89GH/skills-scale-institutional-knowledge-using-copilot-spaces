# OctoAcme — Retrospective & Continuous Improvement

## Purpose
Capture learnings and convert them into actionable improvements.

## When
After each sprint, release, or important milestone. Also after incidents.

## Structure
- What went well
- What could be improved
- Action items (owner, due date)
- Follow-up on previous action items

## Running a Retrospective
- Timebox: 45–75 minutes depending on team size
- Use an anonymous idea board if needed to encourage candor
- Prioritize 2–3 top action items to avoid overload

## Tracking Improvements
- Add action items to the project backlog or issues with clear owners and timelines
- Review outstanding actions in the weekly PM sync

## Retrospective Checklist

| # | Item | Owner | Done? |
|---|---|---|---|
| 1 | Retrospective scheduled within 2 days of sprint/release end | Project Manager | [ ] |
| 2 | All core team members invited (PM, PdM, Tech Lead, Developers, QA, UX, BA) | Project Manager | [ ] |
| 3 | Previous action items reviewed and status updated | Project Manager | [ ] |
| 4 | Team shares "What went well" (async or live) | All | [ ] |
| 5 | Team shares "What could be improved" | All | [ ] |
| 6 | Top 2–3 action items agreed, with owners and due dates | Whole team | [ ] |
| 7 | Action items added to backlog or issue tracker | Project Manager | [ ] |
| 8 | Retrospective notes published to project repo | Project Manager | [ ] |

For role ownership across the project lifecycle see [Role Interactions & RACI](./octoacme-role-interactions-and-raci.md).

## Continuous Improvement Culture
- Measure impact of action items
- Celebrate improvements and make small, iterative changes
